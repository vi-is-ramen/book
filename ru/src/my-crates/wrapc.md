# `wrapc`: Разбор аргументов `rustc` для инструментов `RUSTC_WRAPPER`

## Обзор

Когда вы создаёте инструмент, который перехватывает компиляцию Rust — будь то кэш компиляции, пользовательский профайлер,
статический анализатор или инжектор флагов компоновщика — вам неизбежно понадобится разбирать аргументы командной строки `rustc`.

Это сложнее, чем кажется.

CLI `rustc` — подвижная цель: он смешивает значения с `=` и через пробел, содержит сложный подсинтаксис для компоновки (`-l static:+bundle,+whole-archive=name:renamed`) и постоянно развивается с флагами, доступными только в nightly. Универсальные парсеры CLI, такие как `clap`, слишком жёсткие, слишком тяжёлые и принципиально не подходят для протокола обёрток.

`wrapc` существует именно для решения этой проблемы.

Он предоставляет **строго типизированный, осведомлённый о протоколе парсер**, который:

- Понимает формат вызова Cargo: `<wrapper> - <rustc> <args...>`
- Разбирает сложные флаги в структурированные типы Rust (`--emit`, `--extern`, `-L`, `-l` и т.д.)
- Гарантирует **безупречное восстановление** с помощью `Info::to_args()`
- Корректно обрабатывает неизвестные / nightly-флаги, помещая их в `info.unknown`
- Избегает паник на краевых случаях, таких как разделитель `-` или пропущенные значения

Короче: `wrapc` позволяет вам сосредоточиться на том, что ваша обёртка *делает*, а не на борьбе с разбором аргументов.

## Быстрый старт

Добавьте зависимость:

```bash
cargo add wrapc
```

Затем в вашем `main.rs` обёртки:

```rust
use std::process::Command;
use wrapc::fetch;

fn main() {
    // 1. Разбираем `std::env::args()` в соответствии с протоколом обёртки
    let mut info = fetch().expect("Не удалось разобрать аргументы rustc");

    // 2. Обрабатываем сквозные команды на раннем этапе (help/version/sysroot probes)
    if info.help || info.version || info.print.is_some() {
        let rustc = info.rustc.unwrap_or_else(|| "rustc".to_string());
        let status = Command::new(rustc)
            .args(info.to_args())
            .status()
            .expect("Не удалось запустить rustc");
        std::process::exit(status.code().unwrap_or(1));
    }

    // 3. Анализируем или изменяем контекст компиляции
    if info.crate_name.as_deref() == Some("legacy_crate") {
        info.codegen_opts.push("opt-level=1".to_string());
    }

    // 4. Определяем реальный путь к rustc и восстанавливаем аргументы
    let rustc_path = info.rustc.unwrap_or_else(|| "rustc".to_string());
    let args = info.to_args();

    // 5. Запускаем реальный компилятор
    let status = Command::new(rustc_path)
        .args(&args)
        .status()
        .expect("Не удалось запустить rustc");

    std::process::exit(status.code().unwrap_or(1));
}
```

Это весь скелет. Пять чётких шагов. Никакого разделения строк, никакого хрупкого индексирования, нулевая потеря данных.

## Основные понятия

### Протокол обёртки и разделитель `-`

Cargo вызывает обёртки, используя строгий формат:

```
<wrapper_binary> - <actual_rustc_path> <rustc_args...>
```

`wrapc::fetch()` автоматически:

1. Отбрасывает первые три аргумента
2. Сохраняет реальный путь к компилятору в `info.rustc`
3. Разбирает оставшуюся часть как аргументы `rustc`

#### Краевой случай с `-`

Инструменты иногда передают `-` как имя входного файла, чтобы указать `rustc` читать из `stdin`.
Поскольку протокол обёртки *также* использует `-` как разделитель, наивные парсеры здесь ломаются.

`wrapc` обрабатывает это контекстуально:

```rust
// Вызов: `my_wrapper - rustc -`
let info = wrapc::fetch().unwrap();
assert_eq!(info.rustc, Some("rustc".to_string()));
assert_eq!(info.inputs, vec![std::path::PathBuf::from("-")]);
```

Первый `-` — это разделитель протокола; второй правильно идентифицируется как входной файл.

### Типобезопасный разбор флагов

Вместо возврата `Vec<String>`, `wrapc` разлагает сложные флаги на структурированные типы:

| Флаг | Поле для разбора | Тип |
|------|-------------|------|
| `--emit=llvm-ir,obj` | `info.emit` | `Vec<EmitKind>` |
| `--extern crate=path` | `info.externs` | `Vec<Extern>` |
| `-L kind=path` | `info.libpaths` | `Vec<LibrarySearchPath>` |
| `-l kind:+mods=name:rename` | `info.links` | `Vec<LinkLib>` |

Это позволяет писать логику вроде:

```rust
use wrapc::{LibrarySearchPathKind, LinkLibKind};

for lib_path in &info.libpaths {
    if lib_path.kind == LibrarySearchPathKind::Native {
        println!("Путь для нативного поиска: {:?}", lib_path.path);
    }
}

for link in &info.links {
    if matches!(link.kind, Some(LinkLibKind::Static)) {
        println!("Статическая библиотека: {} (модификаторы: {:?})", link.name, link.modifiers);
    }
}
```

Никаких регулярных выражений. Никакого ручного разделения. Просто типизированные данные.

### Безупречное сквозное восстановление

Обёртка никогда не должна нарушать сборку. Когда вы изменяете подмножество аргументов,
всё остальное должно пройти *точно* так, как задумал Cargo.

`wrapc` гарантирует это через `Info::to_args()`:

- Сохраняет исходные пробелы и формы `=` или через пробел
- Поддерживает порядок аргументов
- Заново выводит неизвестные флаги из `info.unknown` без изменений
- Обрабатывает краевые случаи, такие как значения в кавычках или встроенные пробелы

Вы изменяете то, что нужно; `wrapc` обрабатывает остальное.

### Корректная деградация для nightly-флагов

Rust развивается. Ежедневно в nightly-компиляторах появляются новые флаги.

`wrapc` не притворяется, что знает их все. Вместо этого:

1. Известные флаги -> разбираются в типизированные поля
2. Неизвестные флаги -> сохраняются в `info.unknown` как сырые строки
3. При `to_args()` -> все флаги выводятся заново в точности как были получены

Ваша обёртка не упадёт на новом nightly, и вы молча не потеряете экспериментальные флаги.

## Основные моменты API

### `fetch() -> Result<Info, ParseError>`

Точка входа. Читает `std::env::args()`, разбирает в соответствии с протоколом обёртки
и возвращает строго типизированную структуру `Info`.

### Структура `Info` (выбранные поля)

```rust
pub struct Info {
    // Метаданные протокола
    pub rustc: Option<String>,        // Путь к реальному rustc, если предоставлен

    // Общие флаги
    pub crate_name: Option<String>,
    pub edition: Option<String>,
    pub target: Option<String>,
    pub profile: Option<String>,      // "debug" или "release"

    // Флаги действий
    pub help: bool,
    pub version: bool,
    pub print: Option<String>,        // --print=<kind>

    // Компилируемые единицы
    pub inputs: Vec<PathBuf>,         // Исходные файлы
    pub emit: Vec<EmitKind>,          // --emit=...
    pub out_dir: Option<PathBuf>,

    // Управление зависимостями
    pub externs: Vec<Extern>,         // --extern crate=path
    pub libpaths: Vec<LibrarySearchPath>, // -L kind=path
    pub links: Vec<LinkLib>,          // -l kind:+mods=name:rename

    // Генерация кода и диагностика
    pub codegen_opts: Vec<String>,    // Значения флага -C
    pub cfg: Vec<String>,             // Значения --cfg
    pub features: Vec<String>,        // --cfg feature=...

    // Всё остальное для неизвестных/nightly-флагов
    pub unknown: Vec<String>,
}
```

### `Info::to_args(&self) -> Vec<String>`

Восстанавливает исходный список аргументов без потери данных. Используйте это при передаче аргументов реальному `rustc`.

### `ParseError`

Лёгкий тип ошибки, который указывает, *почему* разбор не удался (например, отсутствует значение, неправильный флаг).
Большинство инструментов-обёрток могут безопасно использовать `.expect()` на `fetch()`, поскольку Cargo гарантирует
правильно сформированные вызовы — но ошибка доступна, если она вам понадобится.

## Продвинутый уровень: изменение контекста компиляции

Поскольку `Info` владеет своими данными, вы можете безопасно изменять поля перед восстановлением:

```rust
// Принудительно устанавливаем опцию codegen для всех крейтов
info.codegen_opts.push("target-cpu=native".to_string());

// Условно добавляем флаг cfg
if info.profile.as_deref() == Some("release") {
    info.cfg.push("feature=\"optimised\"".to_string());
}

// Заменяем путь внешней зависимости
for ext in &mut info.externs {
    if ext.name == "legacy_dep" {
        ext.path = PathBuf::from("/new/path/liblegacy_dep.rlib");
    }
}
```

Все изменения отражаются в выводе `to_args()`.

## Тестирование вашей обёртки

Вам не нужно публиковать или устанавливать глобально для тестирования. Используйте `RUSTC_WRAPPER`:

```bash
# Собираем вашу обёртку
cargo build --release

# Указываем Cargo на ваш бинарный файл
export RUSTC_WRAPPER="$(pwd)/target/release/my_wrapper"

# Запускаем любую команду cargo — ваша обёртка перехватит вызовы rustc
cargo build
```

> **Совет от бывалого**: Всегда проверяйте `info.help`, `info.version` и `info.print` на раннем этапе в вашем `main()`. Если какой-либо из них установлен, немедленно передайте аргументы в `rustc` и выйдите. Это предотвращает вмешательство вашей обёртки во внутренние зонды компилятора Cargo.

## Когда использовать `wrapc`

**Идеален для**:
- Кэшей компиляции (инструменты вроде `sccache`)
- Телеметрии или профилирования во время сборки
- Обёрток статического анализа
- Инжекторов флагов компоновщика или переменных окружения
- Любого инструмента, реализующего `RUSTC_WRAPPER`

**Не предназначен для**:
- Универсального разбора CLI (используйте `clap`, `bpaf` или `argh`)
- Плагинов драйвера `rustc` (используйте API `rustc_driver`)
- Разбора аргументов `cargo` (используйте `cargo_metadata` или специализированные парсеры)
